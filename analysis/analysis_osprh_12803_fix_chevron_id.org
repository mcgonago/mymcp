* Executive Summary

The current implementation of ~ExpandableKeyPairColumn.get_data()~ generates the same ~chevron_id~ for ALL rows in the table, causing only the first row's expand/collapse functionality to work. This is because ~self.creation_counter~ is a class-level attribute that identifies the column instance, not individual rows.

** Critical Problem
Every key pair row receives ~chevron_id = "keypairs_table_chevron1"~, violating HTML ID uniqueness and breaking Bootstrap collapse for all rows except the first.

** Root Cause
The ~datum~ (row data) is not being used in ID generation. The code only uses table-level and column-level identifiers.

** Solution
Use row-specific data from ~datum~ (e.g., keypair name) to generate unique IDs for each row.

* Current Implementation Analysis

** What Each Component Means

*** ~self.table.name~
- Type: String
- Value: ~"keypairs"~ (from ~KeyPairsTable.Meta.name~)
- Scope: *Table-level* - same for all rows
- Purpose: Identifies which table this column belongs to

*** ~self.creation_counter~
- Type: Integer
- Value: Sequential counter assigned when column is created
- Scope: *Column-level* - same for all rows
- Purpose: Distinguishes between multiple column instances in the same table
- Increments: Only when a new Column class instance is created

*** ~datum~
- Type: Object (Keypair instance from OpenStack API)
- Value: The actual keypair data for this specific row
- Scope: *Row-level* - unique for each row
- Available attributes:
  - ~datum.name~ - keypair name (e.g., "test1", "test2", "test3")
  - ~datum.fingerprint~ - unique fingerprint
  - ~datum.type~ - keypair type (e.g., "ssh")
  - ~datum.public_key~ - public key (if available)

* Solutions: How to Generate Unique IDs

** Solution 1: Use Keypair Name (Recommended)

*** Implementation

class ExpandableKeyPairColumn(tables.Column):

from django.utils.http import urlquote

class ExpandableKeyPairColumn(tables.Column):

        return render_to_string(
            "key_pairs/_chevron_column.html",

    def render(self):
        # Generate unique chevron ID using the datum's unique identifier
        object_id = self.table.get_object_id(self.datum)
        chevron_id = "%s_chevron_%s" % (self.table.name, object_id)
        
        return render_to_string(
            "key_pairs/expandable_row.html",
            {"row": self, "chevron_id": chevron_id}
        )

* Key Takeaways

** The Core Problem

~creation_counter~ is a *column-level* attribute, not a *row-level* attribute.

#+BEGIN_SRC text
❌ WRONG: Using column attributes for row identification
✅ RIGHT: Using datum attributes for row identification
#+END_SRC

** The Core Solution

Use ~datum~ (the row data) to generate unique IDs:

#+BEGIN_SRC python
# The datum parameter IS the row-specific data!
def get_data(self, datum):
    object_id = self.table.get_object_id(datum)  # ← Extract unique identifier
    chevron_id = "%s_chevron_%s" % (self.table.name, object_id)
#+END_SRC

** Why get_object_id() Is Best

1. *Already exists* in the table
2. *Already tested* by Horizon
3. *Handles encoding* automatically
4. *Consistent* with how Horizon identifies rows
5. *Respects table logic* for unique identification

** General Principle

When you need a unique identifier per row in a Horizon table:
1. Use ~datum~ (the row object)
2. Extract a unique attribute (~datum.name~, ~datum.id~, etc.)
3. Or use ~table.get_object_id(datum)~ for consistency

* Appendix: Understanding Horizon Table Architecture

** Table Component Hierarchy

#+BEGIN_SRC text
DataTable
├── Meta (configuration)
│   ├── name: "keypairs"
│   ├── row_class: ExpandableKeyPairRow
│   └── ...
├── Columns (defined at class level)
│   ├── chevron: ExpandableKeyPairColumn (creation_counter=1)
│   ├── name: Column (creation_counter=2)
│   ├── key_type: Column (creation_counter=3)
│   └── fingerprint: Column (creation_counter=4)
└── Data (provided at runtime)
    ├── Keypair(name="test1", ...)  ← datum for row 1
    ├── Keypair(name="test2", ...)  ← datum for row 2
    └── Keypair(name="test3", ...)  ← datum for row 3
#+END_SRC

** Rendering Flow

#+BEGIN_SRC text
1. Table.__init__(request, data)
   └─> self.data = [keypair1, keypair2, keypair3]

2. Table.render()
   └─> For each datum in self.data:
       └─> Row(table=self, datum=datum)
           └─> For each column in self.columns:
               └─> cell = Cell(column, row, datum)
                   └─> value = column.get_data(datum)  ← HERE!
                       └─> Cell stores value
#+END_SRC

** Key Insight

~get_data(datum)~ is called *once per cell*, with the *datum for that row*.

This is your opportunity to use row-specific data to generate unique values!

* Summary

** The Problem
Using ~self.creation_counter~ generates the same ID for all rows because it's a column-level counter, not a row-level identifier.

** The Solution
Use ~self.table.get_object_id(datum)~ to get a unique identifier for each row based on the keypair's data.

** The Code Change
#+BEGIN_SRC python
# OLD (broken)
chevron_id = "%s_table_chevron%d" % (self.table.name, self.creation_counter)

# NEW (fixed)
object_id = self.table.get_object_id(datum)
chevron_id = "%s_chevron_%s" % (self.table.name, object_id)
#+END_SRC

** Why It Works
- ~datum~ contains the keypair object for THIS specific row
- ~get_object_id(datum)~ returns the keypair's unique name (URL-encoded)
- Each row gets a different ~datum~, thus a different ~chevron_id~
