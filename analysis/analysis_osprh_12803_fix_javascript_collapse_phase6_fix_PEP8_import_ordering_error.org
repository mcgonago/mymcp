* What is H306?

** OpenStack Hacking Rules

H306 is one of the OpenStack-specific style rules defined in the "hacking" package.

From the OpenStack hacking guidelines:
https://docs.openstack.org/hacking/latest/user/hacking.html

** H306 Definition

#+BEGIN_QUOTE
H306: imports not in alphabetical order

Python imports should be in alphabetical order within their grouping. This helps make
imports easier to scan visually and reduces merge conflicts.
#+END_QUOTE

** Import Grouping

Python imports are grouped in the following order (per PEP8):

1. *Standard library imports* (e.g., =import os=, =from urllib import parse=)
2. *Related third-party imports* (e.g., Django, Flask, etc.)
3. *Local application imports* (e.g., =from horizon import tables=)

Within each group, imports should be *alphabetically sorted*.

** Alphabetical Sorting Rules

The sorting is based on the *full module path*, not just the final component:

#+BEGIN_EXAMPLE
Correct Order (alphabetical by full path):

from django.template.loader import render_to_string  ← "django.template..."
from django import urls                               ← "django.urls"
from django.utils.text import format_lazy            ← "django.utils.text..."
from django.utils.translation import gettext_lazy    ← "django.utils.translation..."

Because:
  "django.template" < "django.urls" < "django.utils.text" < "django.utils.translation"
  (t < u < u.t < u.tr)
#+END_EXAMPLE

* Why Import Order Matters

** Code Readability

Alphabetically sorted imports:
1. Make it easier to scan and find a specific import
2. Reduce the chance of duplicate imports
3. Create a predictable structure

** Merge Conflict Reduction

When multiple developers add imports to the same file:
- Alphabetical order provides a deterministic position for each import
- Reduces conflicts when merging branches
- Makes conflict resolution easier when conflicts do occur

** Consistency Across the Project

OpenStack's Horizon project has thousands of files. Consistent import ordering:
- Makes code reviews faster
- Helps developers move between files more easily
- Enforces project-wide standards

* Common Import Ordering Mistakes

** Example 1: Sorting by Last Component Only

#+BEGIN_EXAMPLE
WRONG:
from django import urls               ← Sorting by "urls"
from django.template.loader import r  ← Sorting by "r" (render_to_string)

RIGHT:
from django.template.loader import r  ← Sorting by full path "django.template..."
from django import urls               ← Sorting by full path "django.urls"
#+END_EXAMPLE

** Example 2: Mixed Standard Library and Third-Party

#+BEGIN_EXAMPLE
WRONG:
from django import urls
import os                             ← Standard library mixed with third-party

RIGHT:
import os                             ← Standard library group first
                                      ← Blank line separator
from django import urls               ← Third-party group second
#+END_EXAMPLE

** Example 3: Inconsistent "from" vs "import"

Both forms should be alphabetically sorted within their group:

#+BEGIN_EXAMPLE
CORRECT:
from django.conf import settings
from django import urls               ← "from django import" is valid
import django.utils                   ← But generally avoid mixing styles

BETTER:
from django.conf import settings
from django import urls
from django import utils              ← Consistent "from X import Y" style
#+END_EXAMPLE

* How Flake8/Hacking Checks Work

** The Check Process

1. *Parser:* Flake8 uses Python's AST (Abstract Syntax Tree) parser
2. *Plugin:* The "hacking" package provides OpenStack-specific rules
3. *H306 Rule:* Checks import statement ordering
4. *Comparison:* Compares full module paths using string comparison

** H306 Implementation

Simplified logic (from the hacking package):

#+BEGIN_SRC python
# Pseudo-code for H306 check
def check_import_order(imports):
    for i in range(len(imports) - 1):
        current = imports[i].module_path
        next = imports[i + 1].module_path
        
        if current > next:  # String comparison
            return "H306: imports not in alphabetical order"
    
    return None  # All good!
#+END_SRC

* Understanding the Full Alphabetical Order

** The Question

Looking at our fixed imports:

#+BEGIN_SRC python
from django.template.loader import render_to_string
from django import urls
from django.utils.text import format_lazy
from django.utils.translation import gettext_lazy as _
from django.utils.translation import ngettext_lazy
#+END_SRC

How is this in alphabetical order? And how are these "grouped"?

** The Answer

These imports are *all in one group* (the "third-party Django imports" group), and they're sorted
alphabetically *by the full module path*:

1. =django.template.loader= ← "template" comes first
2. =django.urls= ← "urls" comes second
3. =django.utils.text= ← "utils.text" comes third
4. =django.utils.translation= ← "utils.translation" comes fourth (twice, same module)

** Why This Order?

Think of it as string comparison of the full module paths:

#+BEGIN_EXAMPLE
"django.template.loader"      (t-e-m-p-l-a-t-e)
  <
"django.urls"                  (u-r-l-s)
  <
"django.utils.text"            (u-t-i-l-s.t-e-x-t)
  <
"django.utils.translation"     (u-t-i-l-s.t-r-a-n-s-l-a-t-i-o-n)
#+END_EXAMPLE

After the common prefix "django.", we compare character by character:
- *'t'* (template) < *'u'* (urls) ← template comes before urls
- *'u'* (urls) < *'u'* (utils), but "urls" < "utils.text" ← urls comes before utils
- *"utils.t[ext]"* < *"utils.t[ranslation]"* ← text comes before translation

** Visual Breakdown

#+BEGIN_EXAMPLE
django
├── template            ← 't' = position 1
│   └── loader
├── urls                ← 'u' = position 2
└── utils               ← 'u' = position 3
    ├── text            ← 'u.t[e]' = position 3a
    └── translation     ← 'u.t[r]' = position 3b
        ├── gettext_lazy
        └── ngettext_lazy
#+END_EXAMPLE

** Detailed String Comparison

Let's verify each comparison step-by-step:

#+BEGIN_EXAMPLE
Comparison 1: "django.template.loader" vs "django.urls"
─────────────────────────────────────────────────────────
After "django.": compare "template" vs "urls"
't' < 'u' → TRUE
Result: template comes first ✓

Comparison 2: "django.urls" vs "django.utils.text"
───────────────────────────────────────────────────
After "django.": compare "urls" vs "utils.text"
'u' == 'u' → continue to next character
'r' vs 't'
'r' < 't' → TRUE
Result: urls comes before utils ✓

Comparison 3: "django.utils.text" vs "django.utils.translation"
───────────────────────────────────────────────────────────────
After "django.utils.": compare "text" vs "translation"
't' == 't' → continue to next character
'e' vs 'r'
'e' < 'r' → TRUE
Result: text comes before translation ✓
#+END_EXAMPLE

** What About Multiple Imports from the Same Module?

The last two imports are from the *same module* (=django.utils.translation=):

#+BEGIN_SRC python
from django.utils.translation import gettext_lazy as _
from django.utils.translation import ngettext_lazy
#+END_SRC

When multiple imports come from the same module, they can appear in any order (or you can
combine them):

#+BEGIN_SRC python
# Option 1: Separate lines (both orderings acceptable)
from django.utils.translation import gettext_lazy as _
from django.utils.translation import ngettext_lazy

# Option 2: Separate lines (reverse order also acceptable)
from django.utils.translation import ngettext_lazy
from django.utils.translation import gettext_lazy as _

# Option 3: Combined (best practice)
from django.utils.translation import gettext_lazy as _, ngettext_lazy
#+END_SRC

The H306 rule only checks that *different modules* are in alphabetical order.

** The "Grouping" Concept

"Within their grouping" refers to the PEP8 import groups:

#+BEGIN_EXAMPLE
Group 1: Standard Library Imports
─────────────────────────────────
from urllib import parse          ← Only standard library import in our file

[blank line separator]

Group 2: Third-Party Imports (All Django imports)
──────────────────────────────────────────────────
from django.template.loader import render_to_string
from django import urls
from django.utils.text import format_lazy
from django.utils.translation import gettext_lazy as _
from django.utils.translation import ngettext_lazy

[blank line separator]

Group 3: Local Application Imports
───────────────────────────────────
from horizon import tables
from openstack_dashboard import api
from openstack_dashboard.usage import quotas
#+END_EXAMPLE

Each group is *independently* sorted alphabetically. You don't sort across groups.

** Quick Summary

In a few sentences:

The imports are alphabetically ordered *by full module path*. After "django.", we compare:
"template" < "urls" < "utils.text" < "utils.translation". The character-by-character comparison
shows: 't' < 'u' (so template before urls), then 'r' < 't' (so urls before utils), then
'e' < 'r' (so text before translation). All these Django imports are in *one group* (third-party
imports), which is alphabetically sorted independently from other groups (standard library and
local imports).
