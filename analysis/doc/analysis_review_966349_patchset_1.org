* Patchset 1
Pulling limitations and summary from all the detail about Patchset 1 (below)

** Current Limitations in Patchset 1
1. *Always Expanded*:  The detail row is always visible; it's not actually collapsible yet
2. *Simple Content*:  Only displays *row.datum* and *row.datum.name* - minimal information
3. *No Styling*:  The detail row has no special CSS classes or styling
4. *No Interactivity*:  No JavaScript to toggle visibility (referenced in docstring but not implemented)

*Note*:  The class docstring mentions "chevron functionality," but this is not implemented in Patchset 1. 
This is likely expanded in later patchsets (particularly Patchset 3 where the template grows from 4 lines to 32 lines).

** Summary
The addition of *row_class = ExpandableKeyPairRow* combined with the custom template creates a mechanism where: 
- Django Horizon's DataTable framework uses the custom row class for rendering
- The custom row class uses a custom template instead of the default
- The custom template includes the normal row rendering AND adds a second *<tr>* element
- The result is that each keypair appears with TWO rows:  the normal data row and an expandable detail row
- This creates the visual "expandable" effect seen in the dashboard, even though in Patchset 1 the rows are always expanded

This forms the foundation for the de-angularization effort, replacing Angular-based expandable rows with a Django template-based approach.

* Detailed Explanation:  Patchset 1 - Expandable Key Pair Rows Implementation
The changes in Patchset 1 introduce a custom row rendering mechanism that displays an additional detail row beneath each key pair entry in the Key Pairs table.
- this creates the expandable row effect seen in the Horizon dashboard.

** Custom Row Class Creation
- *Inherits* from *tables.Row*:  This is the base class for all table rows in Django Horizon's table framework
- *Overrides* the *render()* method*:  Instead of using Horizon's default row rendering, this custom method takes control of how the row HTML is generated
- *Calls *render_to_string()*:  This Django utility function renders the template *key_pairs/expandable_row.html* and returns the resulting HTML as a string
- *Passes context* *"row":  self*:  The row object itself (which contains the keypair data in *self.datum*) is passed to the template for rendering

#+BEGIN_SRC python   # openstack_dashboard/dashboards/project/key_pairs/tables.py

class ExpandableKeyPairRow(tables.Row): 
    """Custom row class for expandable key pair rows with chevron functionality."""
    def render(self): 
        return render_to_string("key_pairs/expandable_row.html",
                                {"row":  self})
#+END_SRC

** Table Configuration
*row_class = ExpandableKeyPairRow*
- this tells Django Horizon's DataTable framework to use the custom *ExpandableKeyPairRow* class instead of the default *tables.Row* class when rendering each row in the table
- *Applies to all rows*:  Every key pair in the table will now be rendered using the custom row class

#+BEGIN_SRC python   # openstack_dashboard/dashboards/project/key_pairs/tables.py

class KeyPairsTable(tables.DataTable): 
    # ... column definitions ...

    class Meta(object): 
        name = "keypairs"
        verbose_name = _("Key Pairs")
        row_class = ExpandableKeyPairRow                                     # ← NEW LINE
        table_actions = (CreateLinkNG, ImportKeyPair, DeleteKeyPairs,
                         KeypairsFilterAction,)
        row_actions = (DeleteKeyPairs,)
#+END_SRC

* Key Concepts

** Django Horizon's DataTable Framework
- *DataTable*:  Container that manages table structure, columns, and rows
- *Row Class*:  Defines how each row is rendered
- *Default Behavior*:  Uses *tables.Row* which renders a single *<tr>* with columns
- *Custom Behavior*:  By setting *row_class*, you can override rendering logic

** Template Inheritance and Inclusion

:  {% include %}:  Inserts another template's content at that point
- *Purpose*:  Reuse the standard row rendering, then add custom content
- *Result*:  Each keypair gets the normal row PLUS the additional detail row

** Context Variables in Template
- *row*:  The ExpandableKeyPairRow object passed from the *render()* method
- *row.datum*:  The actual keypair object from the OpenStack API
- *row.datum.name*:  The name attribute of the keypair object

** HTML colspan Attribute
- *colspan="5"*:  Makes the single *<td>* span across all 5 columns
- *Purpose*:  Creates a full-width detail row beneath the normal row
- *Visual Effect*:  The detail information appears to "expand" below the main row

* Why This Approach?
** Advantages
1. *Non-invasive*:  Reuses existing row rendering via *include*
2. *Maintainable*:  Changes to standard row template automatically apply
3. *Flexible*:  Can add any additional content in the detail row
4. *Framework-compliant*:  Uses Django Horizon's designed extension point (*row_class*)

** Design Pattern
This follows the *Template Method Pattern*: 
- The framework (Django Horizon) defines the structure (DataTable)
- The developer overrides specific methods (render()) to customize behavior
- The result integrates seamlessly with the existing system
