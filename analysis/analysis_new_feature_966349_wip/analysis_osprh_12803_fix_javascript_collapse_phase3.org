** How Bootstrap Makes This Work

*** Bootstrap's Automatic .collapsed Class Management

This is the KEY insight:

When you use ~data-toggle="collapse"~, Bootstrap's JavaScript AUTOMATICALLY:
1. Adds ~.collapsed~ class to the trigger when target is collapsed
2. Removes ~.collapsed~ class when target is expanded
3. Updates this class EVERY time the collapse state changes

This happens automatically - NO custom JavaScript needed!

*** The CSS Selector Pattern

The sidebar uses: ~.collapsed > .openstack-toggle.fa~

Breakdown:
- ~.collapsed~ = the trigger element (when target is collapsed)
- ~>~ = direct child selector
- ~.openstack-toggle.fa~ = the icon span inside the trigger

When collapsed: CSS applies rotation
When expanded: CSS rotation doesn't apply, icon returns to default

** Why This Pattern Is Perfect

1. *Zero Custom JavaScript*
   - Bootstrap handles ALL the class toggling
   - We only write CSS

2. *Smooth Animations*
   - CSS transitions animate the rotation
   - GPU-accelerated transforms

3. *Already Used in Horizon*
   - Proven pattern from the sidebar
   - Consistent with Horizon's design patterns
   - Maintainers are familiar with this approach

4. *Accessible*
   - Bootstrap already manages ~aria-expanded~
   - Screen readers get proper state information

* Technical Deep Dive

** Bootstrap's .collapsed Class Behavior

From Bootstrap 3.4 documentation and source code:

When ~data-toggle="collapse"~ is used, Bootstrap's collapse plugin:

1. *On initialization:*
   - Checks if target has ~.in~ class (visible)
   - If target doesn't have ~.in~, adds ~.collapsed~ to trigger
   - If target has ~.in~, doesn't add ~.collapsed~

2. *On toggle (click):*
   - If expanding: removes ~.collapsed~ from trigger
   - If collapsing: adds ~.collapsed~ to trigger

3. *Events fired:*
   - ~show.bs.collapse~ - fires when starting to expand
   - ~shown.bs.collapse~ - fires when fully expanded
   - ~hide.bs.collapse~ - fires when starting to collapse
   - ~hidden.bs.collapse~ - fires when fully collapsed

4. *ARIA updates:*
   - Bootstrap also updates ~aria-expanded="true/false"~
   - This happens automatically alongside ~.collapsed~ class

** Why We Use :not(.collapsed) Instead of Just .collapsed

Looking at the sidebar, they use:
: .collapsed > .openstack-toggle.fa { rotate(-90deg); }

Why? Because their default icon is ~fa-chevron-down~ (▼):
- When expanded (no ~.collapsed~): default ▼ (points down)
- When collapsed (~.collapsed~): rotate -90deg → ► (points right)

For our case, default icon is ~fa-chevron-right~ (►):
- When collapsed (~.collapsed~): default ► (points right) - no rotation needed
- When expanded (no ~.collapsed~): rotate 90deg → ▼ (points down)

So we use ~:not(.collapsed)~ to apply rotation when expanded.

** CSS Transform vs Changing Icon Class

Two approaches to rotate chevron:

*** Approach 1: CSS Transform (RECOMMENDED)
#+BEGIN_SRC css
.chevron-toggle:not(.collapsed) .fa-chevron-right {
  transform: rotate(90deg);
}
#+END_SRC

Pros:
- Smooth CSS animations
- GPU-accelerated (better performance)
- No DOM manipulation
- No JavaScript needed

Cons:
- Icon is still technically ~fa-chevron-right~, just rotated
- Screen readers don't know about rotation

*** Approach 2: Toggle Icon Classes with JavaScript
#+BEGIN_SRC javascript
$(document).on('shown.bs.collapse', '.collapse', function() {
  $('[data-target="#' + this.id + '"] .fa')
    .removeClass('fa-chevron-right')
    .addClass('fa-chevron-down');
});

$(document).on('hidden.bs.collapse', '.collapse', function() {
  $('[data-target="#' + this.id + '"] .fa')
    .removeClass('fa-chevron-down')
    .addClass('fa-chevron-right');
});
#+END_SRC

Pros:
- Icon class matches visual state
- More semantic

Cons:
- Requires JavaScript
- More complex
- DOM manipulation overhead
- Harder to maintain

** Recommendation: Use Approach 1 (CSS Transform)

Reasons:
1. Matches Horizon's sidebar implementation
2. Zero JavaScript required
3. Better performance
4. Simpler to maintain
5. Horizon already uses this pattern

* Alternative Considered: Using aria-expanded Instead of .collapsed

Could we use ~aria-expanded~ attribute instead of ~.collapsed~ class?

#+BEGIN_SRC css
.chevron-toggle[aria-expanded="true"] .fa-chevron-right {
  transform: rotate(90deg);
}
#+END_SRC

This would work! But:

1. *Bootstrap uses .collapsed as the primary mechanism*
   - ~.collapsed~ is added/removed first
   - ~aria-expanded~ is secondary (for accessibility)

2. *Horizon's pattern uses .collapsed*
   - Consistency with existing codebase
   - Proven pattern

3. *Attribute selectors are slightly slower*
   - Class selectors are faster than attribute selectors
   - Negligible in practice, but why not use the faster one?

**Verdict:** Stick with ~.collapsed~ class for consistency with Horizon.

* Risk Assessment

** Potential Issues

*** Issue 1: Stylesheet Not Loading

*Problem:* CSS file created but not loaded on page.

*Symptoms:* Chevron appears but doesn't rotate.

*Solution:*
- Check browser DevTools → Network tab
- Verify CSS file is loaded
- Check for 404 errors
- May need to add explicit ~<link>~ tag
- Run ~python manage.py collectstatic~
- Clear browser cache

*** Issue 2: CSS Specificity Conflicts

*Problem:* Another CSS rule overriding our rotation.

*Symptoms:* Chevron doesn't rotate or rotates incorrectly.

*Solution:*
- Use browser DevTools → Inspect element
- Check computed styles
- Identify conflicting rule
- Increase specificity if needed:
  : table.datatable .chevron-toggle:not(.collapsed) .fa-chevron-right

*** Issue 3: Transform Not Supported

*Problem:* Old browsers don't support CSS transforms.

*Symptoms:* Chevron doesn't rotate in IE9 or older.

*Solution:*
- Check Horizon's browser support requirements
- If IE9 support needed, add vendor prefixes:
#+BEGIN_SRC css
.chevron-toggle:not(.collapsed) .fa-chevron-right {
  -webkit-transform: rotate(90deg);
  -moz-transform: rotate(90deg);
  -ms-transform: rotate(90deg);
  transform: rotate(90deg);
}
#+END_SRC

*** Issue 4: Icon Appears Blurry After Rotation

*Problem:* Rotated icon looks blurry or pixelated.

*Symptoms:* Visual quality degradation after rotation.

*Solution:*
- Force hardware acceleration:
#+BEGIN_SRC css
.chevron-toggle .fa-chevron-right {
  transform: translateZ(0);  /* Force GPU acceleration */
}
#+END_SRC

** Mitigation Strategies

All of these issues are well-known CSS challenges with documented solutions.
The benefits far outweigh the minimal risk.
