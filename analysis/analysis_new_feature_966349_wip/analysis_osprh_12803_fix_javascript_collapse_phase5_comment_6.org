** The Problem

When the detail row is collapsed, there's a visual gap/spacing that shouldn't be there.
This gap is caused by default table cell padding on the `<td>` element.

Even though the `<div class="collapse">` inside the `<td>` is hidden (height: 0),
the `<td>` element itself still has padding, creating visible space.

* Understanding the CSS

** First CSS Rule: Remove Table Cell Padding

#+BEGIN_SRC css
tr.keypair-detail-row td { padding: 0 };
#+END_SRC

*** What It Does

*Selector:* ~tr.keypair-detail-row td~
- Targets: `<td>` elements that are children of `<tr class="keypair-detail-row">`
- Specificity: Tag + class + tag = medium specificity

*Property:* ~padding: 0~
- Sets all padding (top, right, bottom, left) to 0
- Removes any default or inherited padding from the table cell

*** Why It's Needed

Table cells (`<td>`) typically have default padding (often 8px or similar) applied by:
1. Bootstrap's table styles
2. Horizon's base table styles
3. Browser default styles

When the detail row is collapsed:
- The `<div class="collapse">` inside has `height: 0`
- BUT the `<td>` still has padding
- This creates a visible gap (empty space) between rows

Example visualization:

Before (with default padding):
┌──────────────────────────────────┐
│ test1  (main row)                │
├──────────────────────────────────┤
│ ····                             │ ← Gap from td padding!
│ [collapsed div: height 0]        │
│ ····                             │ ← Gap from td padding!
├──────────────────────────────────┤
│ test2  (main row)                │
└──────────────────────────────────┘

After (padding: 0):
┌──────────────────────────────────┐
│ test1  (main row)                │
├──────────────────────────────────┤
│ [collapsed div: height 0, no gap]│ ← No gap!
├──────────────────────────────────┤
│ test2  (main row)                │
└──────────────────────────────────┘
#+END_EXAMPLE

** Second CSS Rule: Add Margin to Inner Content

#+BEGIN_SRC css
tr.keypair-detail-row td div.keypair-details { margin: 8px };
#+END_SRC

After removing the `<td>` padding, we still want spacing when the details are EXPANDED:

Without margin on the inner div:

#+BEGIN_EXAMPLE

├──────────────────────────────────┤
│ ····                             │ ← Margin (only when expanded)
│ Name: test1                      │
│ Type: ssh                        │
│ Fingerprint: ...                 │
│ ····                             │ ← Margin
└──────────────────────────────────┘
#+END_EXAMPLE

*** Key Insight: Padding vs Margin with Collapse

*Padding* is INSIDE the element:
- Always takes up space (even when element is height: 0)
- Creates unwanted gaps when collapsed

*Margin* is OUTSIDE the element:
- When element has height: 0, margin doesn't create visual space
- Only visible when element is expanded

This is why we:
1. Remove padding from `<td>` (always present)
2. Add margin to inner `<div>` (only visible when expanded)

* Why This Pattern Works

** Bootstrap Collapse Behavior

When Bootstrap collapses an element with `class="collapse"`:

1. *Collapsed state:*
   - Removes ~.in~ class
   - Sets CSS: ~height: 0~
   - Sets CSS: ~overflow: hidden~
   - Element is invisible but still in DOM

2. *Expanded state:*
   - Adds ~.in~ class
   - Animates height from 0 to auto
   - Element becomes visible

** The Problem with Table Cell Padding

Table cells have the CSS box model:

#+BEGIN_EXAMPLE
┌─────────────────────────────────┐
│ ← Padding                       │
│   ┌─────────────────────────┐   │
│   │  Content Area           │   │
│   │  (where div lives)      │   │
│   └─────────────────────────┘   │
│                       Padding → │
└─────────────────────────────────┘
#+END_EXAMPLE

When content has ~height: 0~:
- Content area collapses
- BUT padding remains!
- Result: visible empty space

** The Solution: Padding → Margin Migration

Move spacing from the table cell (padding) to the inner content (margin):

*Table cell:*
- ~padding: 0~ (no permanent spacing)

*Inner content:*
- ~margin: 8px~ (spacing only when content is visible)

When Bootstrap sets the collapse div to ~height: 0~:
- Inner div also effectively has ~height: 0~
- Margin on a zero-height element doesn't create visual space
- Result: no gap!

When Bootstrap expands:
- Inner div has actual height
- Margin creates proper spacing around content
- Result: nice spacing when expanded!

** CSS Margin Collapsing Doesn't Apply Here

Normally, margins can "collapse" (merge) vertically. But in this case:
- The margin is on a div INSIDE a table cell
- Table cells prevent margin collapse
- So the 8px margin works as expected

* Alternative Approaches Considered

** Alternative 1: Use Padding on Collapsible Div

#+BEGIN_SRC css
tr.keypair-detail-row td {
  padding: 0;
}

tr.keypair-detail-row td div.collapse {
  padding: 8px;
}
#+END_SRC

*Problem:* Padding is inside the element that collapses.
When ~height: 0~, the padding is hidden but still part of the collapsed height calculation.
This can cause visual glitches during animation.

*Verdict:* Not recommended.

** Alternative 2: Keep Default Padding, Hide Row When Collapsed

#+BEGIN_SRC css
tr.keypair-detail-row:has(div.collapse:not(.in)) {
  display: none;
}
#+END_SRC

*Problem:*
1. ~:has()~ selector has limited browser support
2. Completely hiding the row is different behavior than collapsing
3. Doesn't work with Bootstrap's collapse animations

*Verdict:* Not compatible with Bootstrap collapse.

** Alternative 3: Negative Margin on Collapse Div

#+BEGIN_SRC css
tr.keypair-detail-row td div.collapse {
  margin: -8px;
}
#+END_SRC

*Problem:* Negative margins are hacky and can cause:
- Content overflow issues
- Unpredictable behavior with animations
- Accessibility problems

*Verdict:* Not a clean solution.

** Why Radomir's Approach is Best

1. *Clean separation of concerns*
   - Cell has no padding (structural)
   - Content has margin (presentational)

2. *Works perfectly with Bootstrap collapse*
   - Margin doesn't create space when height is 0
   - Margin creates space when expanded

3. *Simple and predictable*
   - Easy to understand
   - No hacky tricks
   - Standard CSS patterns

4. *Maintainable*
   - Clear purpose for each rule
   - Easy to adjust spacing (change 8px value)
   - Follows CSS best practices

* CSS Specificity Analysis

** Rule 1: ~tr.keypair-detail-row td { padding: 0; }~

*Specificity:* (0, 1, 2)
- 0 IDs
- 1 class (.keypair-detail-row)
- 2 tags (tr, td)

*Calculated:* 012 (medium specificity)

This should override:
- Bootstrap's default table cell padding
- Horizon's base table styles

May need ~!important~ if stronger rules exist, but try without first.

** Rule 2: ~tr.keypair-detail-row td div.keypair-details { margin: 8px; }~

*Specificity:* (0, 2, 3)
- 0 IDs
- 2 classes (.keypair-detail-row, .keypair-details)
- 3 tags (tr, td, div)

*Calculated:* 023 (higher specificity)

This is very specific and should override most other margin rules.

** If Specificity Issues Arise

If the styles don't apply (checked in DevTools), can increase specificity:

#+BEGIN_SRC css
/* More specific selector if needed */

  margin: 8px;
}
#+END_SRC

Or as last resort, use ~!important~:

#+BEGIN_SRC css

  padding: 0 !important;
}
#+END_SRC

But try without first - the original specificity should be sufficient.

* Conclusion

Radomir's suggestion solves a subtle but important UX issue: the unwanted gap
that appears between rows when details are collapsed.

The solution is elegant:
1. Remove padding from the table cell (structural element)
2. Add margin to the inner content div (presentational element)

Result:
- ✅ No visual gap when collapsed
