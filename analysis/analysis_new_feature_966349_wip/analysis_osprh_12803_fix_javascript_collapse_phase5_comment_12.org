* CSS Specificity Deep Dive

** Understanding Why ~!important~ Was Needed (Patchset 18)

*** Bootstrap's Default Table Cell Padding:
#+BEGIN_SRC css
/* From Bootstrap 3.4.1 */
.table > tbody > tr > td {
  padding: 8px;
  line-height: 1.42857143;
  vertical-align: top;
  border-top: 1px solid #ddd;
}
#+END_SRC

**Specificity:** 0,0,3,4 (3 classes, 4 elements)

*** Our Old Selector (Patchset 18):
#+BEGIN_SRC css
tr.keypair-detail-row td {
  padding: 0 !important;
}
#+END_SRC

**Specificity:** 0,0,1,2 (1 class, 2 elements)

**Result:** Our selector LOSES to Bootstrap's, so we needed ~!important~ 😢

---

** How Radomir Fixed It (Patchset 19)

*** Our New Selector:
#+BEGIN_SRC css
.table > tbody > tr.keypair-detail-row > td {
  padding: 0;
}
#+END_SRC

**Specificity:** 0,0,2,4 (2 classes, 4 elements)

**Result:** Our selector WINS against Bootstrap's naturally! 🎉

** Specificity Calculation Chart

| Selector | IDs | Classes | Elements | Total | Winner? |
|----------+-----+---------+----------+-------+---------|
| Bootstrap: ~.table>tbody>tr>td~ | 0 | 1 | 3 | 0,0,1,3 | ❌ |
| Our Old: ~tr.keypair-detail-row td~ | 0 | 1 | 2 | 0,0,1,2 | ❌ (loses to Bootstrap) |
| Our New: ~.table>tbody>tr.keypair-detail-row>td~ | 0 | 2 | 4 | 0,0,2,4 | ✅ (beats Bootstrap!) |

** Key Takeaway

~!important~ is usually a code smell indicating insufficient CSS specificity.

Better approach:
1. Match or exceed the specificity of the rule you're overriding
2. Use the same selector structure (child selectors ~>~)
3. Add more specific classes

* Key Technical Insights

** 1. CSS Specificity is More Maintainable Than ~!important~

~!important~ is a brute-force override that:
- Makes CSS harder to debug
- Creates specificity wars (everything needs ~!important~)
- Indicates poor selector design
- Can't be overridden (except by another ~!important~)

Better approach: Match the specificity of what you're overriding.

---

** 2. Negative Margins for Overlapping Borders

Instead of removing borders entirely, use negative margins to overlap them:

#+BEGIN_SRC css
/* Old approach: Remove border */
border-top: 0;

/* New approach: Overlap borders */
margin-top: -1px;
#+END_SRC

This keeps proper visual separation when expanded while eliminating double-lines when collapsed.

---

** 3. Child Selectors (~>~) vs Descendant Selectors

*Descendant selector (space):*
#+BEGIN_SRC css
.parent .child { }  /* Any descendant */
#+END_SRC

*Child selector (~>~):*
#+BEGIN_SRC css
.parent > .child { }  /* Direct child only */
#+END_SRC

Child selectors are:
- More specific
- More performant (browser doesn't traverse entire tree)
- More predictable (won't accidentally match nested elements)

---

** 4. Remove Defensive CSS Rules

When debugging CSS, we often add "defensive" rules like:
#+BEGIN_SRC css
height: 0 !important;
margin: 0 !important;
padding: 0 !important;
border: 0 !important;
line-height: 0;
#+END_SRC

But these are often unnecessary. Radomir's approach:
1. Test with minimal CSS
2. Only add rules that solve actual problems
3. Trust the browser's default behavior

---

** 5. Matching Selector Structure with Libraries

When overriding library CSS (Bootstrap, etc.):
- Look at the library's selector structure
- Match or exceed it
- Use the same child/descendant patterns
- Bootstrap uses ~.table>tbody>tr>td~, so we use ~.table>tbody>tr.keypair-detail-row>td~

*** CSS Specificity Calculation
#+BEGIN_EXAMPLE
Specificity = (inline-styles, IDs, classes/attributes, elements)

  = (0, 0, 2, 4)  ← .table + .keypair-detail-row = 2 classes

#+END_EXAMPLE

*** When to Use ~!important~ (Almost Never)
- ✅ Utility classes that should always win (e.g., ~.hidden { display: none !important; }~)
- ✅ Overriding inline styles (which have highest specificity)
- ❌ Because your selector isn't specific enough (fix the selector instead!)
