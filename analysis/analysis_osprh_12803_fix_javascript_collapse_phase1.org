* Executive Summary

Radomir Dopieralski (project maintainer) commented on patchset 7 that the chevron_id generation logic is duplicated between ~ExpandableKeyPairRow~ and ~ExpandableKeyPairColumn~, creating a risk of introducing inconsistencies.

** Recommendation
Extract the duplicated logic into a helper function ~get_chevron_id(table, datum)~ that both classes can call.

** Benefits
- ✅ DRY principle (Don't Repeat Yourself)
- ✅ Single source of truth for ID generation
- ✅ Eliminates risk of divergence between the two implementations
- ✅ Easier to maintain and test

** The Problem

These two lines are **exactly the same** in both places:
#+BEGIN_SRC python
object_id = self.table.get_object_id(datum)
chevron_id = "%s_chevron_%s" % (self.table.name, object_id)
#+END_SRC

** Why This Is A Problem

*** 1. Code Duplication (DRY Violation)
- Same logic written twice
- Violates "Don't Repeat Yourself" principle

*** 2. Maintenance Risk
If we need to change the ID format, we must remember to update **both** places:
- Change in ~ExpandableKeyPairRow~ but forget ~ExpandableKeyPairColumn~ → Bug!
- Different developers might update only one → Inconsistency!

*** 3. Testing Complexity
- Must test the same logic in two different contexts
- No single test point for ID generation

*** 4. Future Changes
If someone wants to add:
- Prefix/suffix to IDs
- Different encoding scheme
- Debugging info to IDs
- Namespace separation

They must update two locations and ensure consistency.

** Example Scenario: What Could Go Wrong

Imagine a future developer wants to add a prefix for namespacing:

*Developer updates ExpandableKeyPairColumn:*
#+BEGIN_SRC python
chevron_id = "kp_%s_chevron_%s" % (self.table.name, object_id)  # Added "kp_" prefix
#+END_SRC

*But forgets to update ExpandableKeyPairRow:*
#+BEGIN_SRC python
chevron_id = "%s_chevron_%s" % (self.table.name, object_id)  # No prefix!
#+END_SRC

*Result:*
- Chevron link: ~href="#kp_keypairs_chevron_test1"~
- Detail row: ~id="keypairs_chevron_test1"~
- IDs don't match!
- ❌ Expand/collapse broken again!

* Proposed Solution

** Extract to Helper Function

Create a module-level helper function that encapsulates the ID generation logic.

*** Function Definition

#+BEGIN_SRC python
def get_chevron_id(table, datum):
    """Generate a unique chevron ID for expandable key pair rows.
    
    This function ensures consistent ID generation between the chevron toggle
    column and the expandable detail row. The ID is based on the table name
    and the unique object identifier for the datum.
    
    Args:
        table: The DataTable instance (provides table.name and get_object_id())
        datum: The key pair object (passed to get_object_id())
    
    Returns:
        str: A unique chevron ID in the format "{table_name}_chevron_{object_id}"
        
    Example:
        >>> table.name = "keypairs"
        >>> datum.name = "test1"
        >>> get_chevron_id(table, datum)
        'keypairs_chevron_test1'
    """
    object_id = table.get_object_id(datum)
    return "%s_chevron_%s" % (table.name, object_id)
#+END_SRC

* Design Considerations

** Why Module-Level Function?

*** Option 1: Module-Level Function (Chosen)
#+BEGIN_SRC python
def get_chevron_id(table, datum):
    ...

class ExpandableKeyPairRow(tables.Row):
    def render(self):
        chevron_id = get_chevron_id(self.table, self.datum)
#+END_SRC

*Advantages:*
- ✅ Clean separation of concerns
- ✅ Not tied to any specific class
- ✅ Easy to test independently
- ✅ Clear function signature
- ✅ Can be imported/reused elsewhere if needed

*Disadvantages:*
- ⚠️ Adds another name to module namespace (minor)

*** Option 2: Static Method on One Class
#+BEGIN_SRC python
class ExpandableKeyPairRow(tables.Row):
    @staticmethod
    def get_chevron_id(table, datum):
        ...
    
    def render(self):
        chevron_id = ExpandableKeyPairRow.get_chevron_id(self.table, self.datum)
#+END_SRC

*Advantages:*
- ✅ Keeps function within class scope

*Disadvantages:*
- ❌ Arbitrary choice of which class to put it in
- ❌ Implies it "belongs" to ~ExpandableKeyPairRow~ (it doesn't)
- ❌ Other class must reference it via class name
- ❌ Less intuitive

*** Option 3: Method on Table Class
#+BEGIN_SRC python
class KeyPairsTable(tables.DataTable):
    def get_chevron_id(self, datum):
        ...
#+END_SRC

*Advantages:*
- ✅ Accessible via ~self.table.get_chevron_id(datum)~

*Disadvantages:*
- ❌ Pollutes table class with row-specific logic
- ❌ Tightly couples table to this specific rendering approach
- ❌ Doesn't make sense for tables that don't have expandable rows

*** Conclusion
Module-level function is the cleanest approach for this use case.

** Function Signature Design

*** Parameters
#+BEGIN_SRC python
def get_chevron_id(table, datum):
#+END_SRC

*Why not just ~datum~?*

We need ~table.name~ and ~table.get_object_id(datum)~, so we need the table instance.

*Why explicit parameters?*

Makes dependencies clear:
- Needs access to table configuration (~table.name~)
- Needs access to table's object ID logic (~table.get_object_id()~)
- Needs the datum to identify the specific row

*** Return Value
#+BEGIN_SRC python
return "%s_chevron_%s" % (table.name, object_id)
#+END_SRC

Returns a string in the format: ~{table_name}_chevron_{object_id}~

Examples:
- ~"keypairs_chevron_test1"~
- ~"keypairs_chevron_test2"~
- ~"keypairs_chevron_my%20keypair"~ (URL-encoded name)

** Placement in File

The function should be placed **before** the classes that use it:
