**Connection to Original:**

* Executive Summary

The column width shifting issue occurs because Bootstrap tables use ~table-layout: auto~ by default, which recalculates column widths based on content. When the detail row expands/collapses (changing the table's total height), the browser's table layout algorithm may redistribute column widths slightly.

The AngularJS version likely uses ~table-layout: fixed~ or explicit column widths to prevent this shifting.

**Solution:** Add ~table-layout: fixed~ to the Key Pairs table in the CSS. This forces columns to maintain their initial widths regardless of content changes, just like the AngularJS version.

**Impact:** One line of CSS fixes the issue completely.

* Detailed Findings

** Understanding Table Layout Algorithms

*** ~table-layout: auto~ (Bootstrap's Default)

#+BEGIN_SRC css
/* Bootstrap 3 default behavior */
.table {
  table-layout: auto; /* Implicit default */
}
#+END_SRC

**How it works:**
1. Browser calculates column widths based on content
2. Each column gets width based on its longest content
3. When content changes (table height changes), widths may be recalculated
4. Algorithm is complex and can cause subtle shifts

**Pros:**
- Automatically fits content
- Good for variable-width content
- Responsive to content changes

**Cons:**
- Columns can shift when content changes
- Performance cost (recalculation on every change)
- Unpredictable width distribution

---

*** ~table-layout: fixed~ (What We Need)

#+BEGIN_SRC css
/* Force stable column widths */
.table {
  table-layout: fixed;
}
#+END_SRC

**How it works:**
1. Column widths calculated based on FIRST row only
2. Once calculated, widths never change (even if content changes)
3. Remaining rows must fit into these fixed widths
4. Much faster rendering (no recalculation)

**Pros:**
- ✅ Columns never shift (stability)
- ✅ Better performance
- ✅ Predictable layout
- ✅ Matches AngularJS behavior

**Cons:**
- Content may overflow if too wide (but we handle this with text wrapping)
- Less flexible for truly variable content

---

** Why Column Shifting Happens

*** The Collapse Animation Triggers Reflow

When Bootstrap's collapse expands/collapses:

#+BEGIN_EXAMPLE
1. User clicks chevron
2. Bootstrap starts collapse animation (height change)
3. Detail row height changes from 0 → full height (or vice versa)
4. Table's total height changes
5. Browser says: "Table height changed, let me recalculate column widths!"
6. Column widths shift by 1-3 pixels
7. User sees visual "jitter"
#+END_EXAMPLE

*** The Detail Row Uses Colspan

Our detail row spans all columns:

#+BEGIN_SRC html
<tr class="keypair-detail-row">
  <td colspan="{{ row.cells|length }}">
    <!-- Detail content -->
  </td>
</tr>
#+END_SRC

This colspan cell doesn't contribute to column width calculation (it's flexible), but when it expands/collapses, it changes the table's overall dimensions, triggering recalculation in ~table-layout: auto~ mode.

** Browser Table Layout Algorithm Details

*** How ~table-layout: auto~ Works

1. Browser reads all table content
2. Calculates minimum width needed for each column
3. Distributes available space proportionally
4. Recalculates on ANY content or height change
5. Complex algorithm (considers text length, wrapping, etc.)

*** How ~table-layout: fixed~ Works

1. Browser reads ONLY the first row
2. Distributes space equally (or based on col widths)
3. Never recalculates (locked forever)
4. Simple algorithm (fast!)

* FINAL Q&A: Understanding the CSS Selector Pattern

** Question from User

"Can you please explain the double lines that are seen in the solution?"

#+BEGIN_SRC css
table#keypairs th:nth-child(1),
table#keypairs td:nth-child(1),
table#keypairs th:nth-child(2),
table#keypairs td:nth-child(2) {
  width: 1%;
  white-space: nowrap;
}
#+END_SRC

Why do we target both ~th~ and ~td~ for each column?

** Answer: Targeting Both Header and Data Cells

*** What ~th~ and ~td~ Mean

- ~th~ = **Table Header** cells (the column titles in ~<thead>~)
  - Examples: "Key Pair Name", "Key Pair Type", "Fingerprint", "Actions"
  - These are the bold column titles at the top of the table

- ~td~ = **Table Data** cells (the actual data rows in ~<tbody>~)
  - Examples: "test1", "ssh", "83:9d:ce:a7:...", etc.
  - These are the regular cells containing key pair information

*** Why We Need BOTH

HTML tables have a specific structure:

#+BEGIN_SRC html
<table id="keypairs">
  <thead>
    <tr>
      <th><!-- Chevron header --></th>
      <th>Key Pair Name</th>
      <th>Key Pair Type</th>
      <th>Fingerprint</th>
      <th>Actions</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><!-- Chevron icon --></td>
      <td>test1</td>
      <td>ssh</td>
      <td>83:9d:ce:a7:...</td>
      <td><!-- Delete button --></td>
    </tr>
    <tr>
      <td><!-- Chevron icon --></td>
      <td>test2</td>
      <td>x509</td>
      <td>a1:b2:c3:d4:...</td>
      <td><!-- Delete button --></td>
    </tr>
  </tbody>
</table>
#+END_SRC

*** Three Critical Reasons

**** 1. Column Width Consistency

Browsers calculate column widths based on **BOTH header and data cells**. If we only style one:

- Style only ~th~: Data cells might ignore the width
- Style only ~td~: Header cells might be wider, causing misalignment

**We must style BOTH to ensure headers and data columns have the same width.**

**** 2. Browser Rendering Algorithm

The CSS table layout algorithm considers:
1. Content width in header cells (~th~)
2. Content width in data cells (~td~)
3. Explicit width declarations on both

Without styling both, the browser picks the **maximum** width needed, potentially breaking our tight spacing goal.

**** 3. Inheritance Doesn't Work Here

CSS ~width~ is NOT inherited from parent to child. Setting width on ~<table>~ doesn't automatically apply to ~<th>~ or ~<td>~.

**Each cell type must be explicitly targeted.**

*** The Pattern Explained Line-by-Line

#+BEGIN_SRC css
table#keypairs th:nth-child(1),  /* Header cell in column 1 (Chevron) */
table#keypairs td:nth-child(1),  /* Data cells in column 1 (Chevron) */
table#keypairs th:nth-child(2),  /* Header cell in column 2 (Name) */
table#keypairs td:nth-child(2) { /* Data cells in column 2 (Name) */
  width: 1%;           /* Force minimum width */
  white-space: nowrap; /* Prevent wrapping */
}
#+END_SRC

This is a **CSS selector list** using commas to apply the same styles to multiple selectors:
- All four selectors share the same declaration block ~{ width: 1%; white-space: nowrap; }~
- This ensures column 1 (Chevron) is tight in BOTH header and data
- This ensures column 2 (Name) is tight in BOTH header and data

*** Visual Comparison

*Without targeting ~th~:*
#+BEGIN_EXAMPLE
| Chevron (wide header) | Name (wide header) | Type | Fingerprint | Actions |
|-------------|---------|------|-------------|---------|
| ▶ (tight)   | test1   | ssh  | 83:9d:...   | Delete  |
#+END_EXAMPLE
Headers would be wider than data cells → **misaligned columns** ❌

*With targeting both ~th~ and ~td~:*
#+BEGIN_EXAMPLE
| ▶ | Name  | Type | Fingerprint      | Actions |
|---|-------|------|------------------|---------|
| ▶ | test1 | ssh  | 83:9d:ce:a7:...  | Delete  |
| ▶ | test2 | x509 | a1:b2:c3:d4:...  | Delete  |
#+END_EXAMPLE
Headers and data have same width → **perfect alignment** ✅

*** Summary

| Element | Purpose | Why Both? |
|---------|---------|-----------|
| ~th~ | Style header cells | Headers control column width |
| ~td~ | Style data cells | Data also controls column width |
| Both | Consistent width | Browser needs both to match |

**TL;DR:** CSS requires styling both ~<th>~ and ~<td>~ to ensure headers and data cells have identical widths, preventing column misalignment.
